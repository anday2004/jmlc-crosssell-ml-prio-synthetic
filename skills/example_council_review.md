# Примеры работы ИИ-навыка `jmlc-adaption-council`

Ниже два реальных прогона навыка по этому публичному репозиторию. Цель —
показать не абстрактное «ИИ помог», а конкретный рабочий цикл: вход → находки
по линзам ревью → внесенное исправление. Оба примера построены на открытых
артефактах и не используют рабочие данные.

Формат вывода навыка задан в [`jmlc-adaption-council/SKILL.md`](jmlc-adaption-council/SKILL.md):
`Decision / Findings (P0–P2) / Chairman Summary / Next Steps`.

---

## Пример 1. Engineering-линза: заявка на MLOps не подтверждена кодом

**Вход:** `Use $jmlc-adaption-council to review the engineering claims in the article against the public repo.`

**Вывод навыка:**

```markdown
Decision: needs changes

Findings:
- P0 (Engineering): Раздел «Инженерная часть» статьи заявляет CI, Docker-образ и
  Airflow-оркестрацию, но в публичном репозитории нет ни Dockerfile, ни workflow
  для CI, ни DAG. Критерий №1 конкурса (Git, Docker, CI/CD, MLOps) проверяется
  по артефактам — ревьюер увидит claim без воспроизводимого подтверждения.
- P1 (Engineering): Четыре способа запуска (notebook, run_experiment.py, demo.py)
  частично дублируются; demo.py — тонкая обертка над run_experiment.py.
- P2 (NDA): Продакшн-стек (GitLab/Airflow) корректно описан словами и под NDA;
  это нормально, но в публичной версии нужна облегченная, реально запускаемая
  параллель, иначе claim голословен.

Chairman Summary:
Сильная DS-часть (uplift, IPS/SNIPS) контрастирует с инженерной витриной:
рабочий MLOps есть только в закрытом контуре, а публичный артефакт его не
показывает. Это единственный системный разрыв между статьей и кодом.

Next Steps:
1. Добавить реально собираемый Dockerfile, который ставит зависимости и
   прогоняет пайплайн + тесты.
2. Добавить GitHub Actions (pytest на каждый push) и бейдж в README.
3. Добавить иллюстративный Airflow DAG поверх тех же step-функций; пометить,
   что продакшн-DAG живет в закрытом контуре.
4. Свести запуск к трем входам: notebook / run_experiment.py / dag.
```

**Что исправлено по итогам:**

- добавлены [`Dockerfile`](../Dockerfile) и [`.github/workflows/ci.yml`](../.github/workflows/ci.yml) (тесты + smoke-прогон + сборка образа);
- добавлен [`dags/crosssell_dag.py`](../dags/crosssell_dag.py) — реальная декомпозиция `generate → score → policies → evaluate` на тех же функциях из `utils/`;
- удален `demo.py`, оставлены три входа (notebook / `run_experiment.py` / dag);
- в статью добавлено явное разделение: продакшн-MLOps под NDA против облегченной публичной параллели.

## Пример 2. NDA-линза: имя работодателя не должно утекать в публичную версию

**Вход:** `Use $jmlc-adaption-council to diff the contest-submission article against the public repo article for NDA leaks.`

**Вывод навыка:**

```markdown
Decision: needs changes

Findings:
- P0 (NDA): Конкурсная (сдаваемая) версия статьи содержит прямое имя
  работодателя в бизнес-постановке: «...в финтех-экосистеме Т-Банка...».
  Такая формулировка допустима только в закрытой сдаваемой версии и НЕ должна
  попадать в публичный GitHub-репозиторий.
- P1 (NDA): Нужна явная пометка, что публичная и сдаваемая версии статьи
  различаются, чтобы их случайно не перепутать при коммите.

Chairman Summary:
Граница NDA проходит ровно между двумя версиями одного текста. Публичная
версия должна обобщать работодателя до «крупной финтех-экосистемы», иначе при
пуше в открытый репозиторий имя компании утечет вместе с описанием внутренней
задачи.

Next Steps:
1. В публичной версии статьи заменить «Т-Банка» на «крупной финтех-экосистеме».
2. Оставить точную формулировку только в сдаваемой версии вне публичного репо.
3. Добавить в шапку публичной версии пометку «Не совпадает со статьей на конкурс».
4. Прогнать grep по публичному репо на имена работодателя/брендов перед пушем.
```

**Что исправлено по итогам:**

- в публичной версии статьи ([`article/crosssell_extra_npv_public.tex`](../article/crosssell_extra_npv_public.tex)) работодатель обобщен до «крупной финтех-экосистемы»; точная формулировка осталась только в сдаваемой версии вне репозитория;
- в шапку публичной версии добавлена пометка «Не совпадает со статьей на конкурс JMLC»;
- grep по публичному репо на имена работодателя и брендов проходит чисто (см. [`docs/nda_boundary.md`](../docs/nda_boundary.md)).

---

Оба примера показывают один и тот же принцип применения ИИ в проекте: навык не
принимает решений за автора, а структурно проверяет артефакт по фиксированным
линзам (ML, causal, product, engineering, NDA), формулирует находки с
приоритетами и предлагает конкретные правки, которые автор затем вносит вручную.
