# Контракт исторических событий

Этот документ описывает, как должны быть устроены исторические события, чтобы один и тот же синтетический датасет подходил для:

- табличных моделей отклика;
- mini-transformer encoder по пользовательским последовательностям;
- uplift-моделирования на рандомизированных показах;
- оффлайн-оценки стратегий без утечки таргета.

Контракт специально сделан безопасным для публичного репозитория: здесь нет внутренних названий таблиц, реальных продуктов, точных временных окон, настоящих категорий или производственной логики.

## 1. Гранулярность решения

Единица принятия решения:

```text
user_id, period_id
```

Для каждой такой единицы стратегия видит:

- историю пользователя, доступную до `decision_time`;
- набор доступных CrossSell-категорий;
- категориальные значения, например нормированный `NPV`;
- продуктовые ограничения, например разнообразие типов категорий;
- рандомизированные логи показов для оффлайн-оценки.

Гранулярность обучения и скоринга:

```text
user_id, period_id, category_id
```

Такая строка отвечает на вопрос: что произойдет, если эту категорию показать этому пользователю в этом периоде?

## 2. Таблица исторических событий

Таблица событий должна содержать одну строку на одно событие пользователя до даты принятия решения.
Общий формат соответствует последовательностной витрине:

```text
user_id - event_time - attribute_1 - ... - attribute_N
```

В публичной версии конкретные `attribute_*` раскрываются как обезличенные поля
ниже. В рабочем контуре они могут соответствовать продуктам, сервисам,
категориям, экранам или другим сущностям, но реальные названия и схемы не
переносятся в репозиторий.

| Колонка | Тип | Смысл |
|---|---:|---|
| `user_id` | int | обезличенный синтетический ключ пользователя |
| `event_time` | int или datetime | время события или относительное время внутри периода |
| `event_family` | категория | крупный тип события: клик, транзакция, действие в приложении, прошлое предложение, жизненный цикл продукта |
| `event_type` | категория | конкретный токен события внутри крупного типа |
| `entity_type` | категория | сущность события: категория, сервис, группа продавцов, экран, продукт |
| `entity_id` | категория/int | синтетический идентификатор сущности |
| `category_id` | int или null | CrossSell-категория, если событие с ней связано |
| `group_id` | категория или null | группа разнообразия, например подписочный, страховой или счетовый тип |
| `amount_bucket` | категория или null | бакет суммы транзакции, но не сырая сумма |
| `channel` | категория | синтетическая поверхность: приложение, веб, пуш, витрина, другое |
| `event_value` | float | ограниченная синтетическая интенсивность, например бакет длительности или транзакционный скор |
| `period_id` | int | период, в котором произошло событие |

В публичной версии не должно быть сырых идентификаторов, точных временных меток, названий продавцов, настоящих продуктов, внутренних экранов или точных сумм.

## 3. Типы событий

### Клики

Клики описывают легкие сигналы интереса:

- предложение было показано в предыдущем периоде;
- пользователь открыл карточку категории;
- пользователь посмотрел подробности;
- пользователь нажал целевую кнопку;
- пользователь скрыл или проигнорировал предложение.

Для трансформера такие события помогают отличать пользователей, которые реагируют на CrossSell-поверхности, от пользователей с высокой фоновой склонностью к утилизации. Для защиты от утечки разрешены только события строго до текущего решения.

### Транзакции

Транзакции описывают поведенческий контекст:

- транзакция в синтетической группе продавцов;
- повторяющийся платеж;
- бакет входящего платежа;
- сберегательная или инвестиционная активность;
- высокоуровневая категория расходов.

Все транзакционные признаки должны быть забакечены и обезличены. Цель не в том, чтобы восстановить реальные операции, а в том, чтобы создать последовательные сигналы, влияющие на базовую вероятность утилизации и чувствительность к показу.

### Действия в приложении

Действия в приложении описывают навигацию и вовлеченность:

- пользователь открыл группу экранов;
- пользователь воспользовался поиском;
- пользователь перешел в продуктовый раздел;
- пользователь открыл документ или выписку;
- пользователь совершил справочное или сервисное действие.

Эти события полезны для трансформера, потому что порядок имеет значение: недавний поиск после просмотра продуктового раздела должен давать другой сигнал, чем те же события, разнесенные далеко во времени.

### Прошлые предложения

Прошлые CrossSell-события описывают историю взаимодействия с предложениями:

- категория была показана в предыдущем периоде;
- категория была выбрана или активирована;
- категория была утилизирована;
- предложение было проигнорировано;
- недавно показывалась похожая категория.

Эти события кодируют усталость от повторных показов, новизну и повторный интерес. Важно: история не должна включать текущий рандомизированный показ и будущую утилизацию.

### Жизненный цикл продукта

События жизненного цикла описывают изменения состояния:

- синтетический продукт был открыт до решения;
- синтетический продукт был закрыт до решения;
- подписочный сервис был начат или остановлен;
- изменилось состояние доступности.

Такие события влияют на доступность категорий и могут входить в признаки, если они известны до даты решения.

## 4. Правила защиты от утечки

Для строки `(user_id, period_id, category_id)` история строится по правилу:

```text
event_time <= decision_time(period_id) - data_lag
```

В последовательность нельзя включать:

- текущий показ;
- текущий выбор категории;
- текущую утилизацию;
- события после рандомизированного назначения;
- таргет или прямые прокси таргетного окна;
- будущие изменения доступности.

Разделение данных должно быть временным:

```text
train periods < calibration periods < validation periods < test periods
```

Случайное разбиение строк не подходит: будущие действия того же пользователя могут протечь в прошлое.

## 5. Вход mini-transformer encoder

Для каждой пары `user_id, period_id` строится последовательность фиксированной длины:

```text
E_i = [event_1, event_2, ..., event_T]
```

События сортируются по времени. Если история слишком длинная, берутся последние `max_seq_len` событий. Если история короткая, слева добавляется padding и строится маска внимания.

Каждое событие представляется набором эмбеддингов и числовых признаков:

| Поле | Кодирование |
|---|---|
| `event_family` | эмбеддинг |
| `event_type` | эмбеддинг |
| `entity_type` | эмбеддинг |
| `entity_id` | эмбеддинг с синтетическим словарем |
| `category_id` | эмбеддинг, включая null-бакет |
| `group_id` | эмбеддинг, включая null-бакет |
| `amount_bucket` | эмбеддинг, включая null-бакет |
| `channel` | эмбеддинг |
| давность до решения | эмбеддинг временного бакета |
| позиция в последовательности | позиционный эмбеддинг |
| `event_value` | небольшая числовая проекция |

Mini-transformer encoder возвращает представление пользователя:

```text
h_i = MiniTransformer(E_i)
```

В публичной реализации encoder обучает event embeddings, attention pooling и
supervised head на train-периодах. После этого поверх последовательностного
представления обучается outcome-модель с флагом показа.

Для каждой кандидатной категории `k` объединяются:

```text
h_i, category_embedding_k, h_i * category_embedding_k, |h_i - category_embedding_k|
```

Затем ранжирующая голова предсказывает:

```text
p(utilization = 1 | user history, category, shown = z)
```

Для uplift-моделирования используется S-learner:

- одна outcome-модель получает признаки пары и флаг `shown`;
- для той же пары модель запускается при `shown = 1` и `shown = 0`;
- uplift равен `p1 - p0`;
- extra NPV равен `NPV_k * (p1 - p0)`.

## 6. Логика генерации синтетики

Генератор должен создавать скрытые свойства пользователя:

- общая активность;
- продуктовая зрелость;
- предпочтения по типам категорий;
- чувствительность к рекомендациям;
- усталость от повторных предложений;
- интенсивность транзакций.

У категорий должны быть:

- `category_id`;
- `group_id`;
- нормированный `NPV`;
- скрытый эмбеддинг;
- частота доступности;
- базовая популярность;
- чувствительность к показу.

События сэмплируются из взаимодействия пользовательских свойств и свойств категорий. Затем истинные потенциальные исходы генерируются через:

```text
base_score(user, category, history) -> p0
treatment_effect(user, category, history) -> p1 - p0
```

У трансформера должен быть настоящий последовательностный сигнал:

- недавние клики повышают краткосрочный интерес;
- транзакции в связанных синтетических группах меняют базовую склонность;
- прошлые проигнорированные предложения снижают чувствительность;
- прошлая связанная утилизация меняет доступность и будущий интерес;
- действия в приложении указывают на активное изучение продуктов;
- временные расстояния важны, а не только счетчики событий.

## 7. Табличные Признаки Из Тех Же Событий

Табличный baseline должен использовать агрегаты из тех же событий, с тем же отсечением по времени:

- счетчики по типам событий в последних окнах;
- давность последнего связанного события;
- число связанных кликов;
- число прошлых показанных, проигнорированных и утилизированных предложений;
- транзакционные бакеты по синтетическим группам;
- интенсивность действий в приложении;
- склонность к типу категории;
- признаки усталости;
- флаги доступности.

Так сравнение остается честным: трансформер получает последовательности, бустинг получает агрегаты, и обе модели используют только информацию, доступную до даты решения.

## 8. Рандомизированные Назначения

Синтетическая таблица рандомизированных показов должна хранить:

| Колонка | Смысл |
|---|---|
| `user_id` | ключ пользователя |
| `period_id` | период решения |
| `category_id` | кандидатная категория |
| `is_available` | категория доступна пользователю |
| `shown` | индикатор рандомизированного показа |
| `propensity` | вероятность наблюденного назначения |
| `utilized` | наблюдаемый результат в таргетном окне |
| `p0_true` | синтетическая истинная вероятность без показа |
| `p1_true` | синтетическая истинная вероятность с показом |
| `extra_npv_true` | значение для оракула: `NPV * (p1_true - p0_true)` |

Рандомизация должна соблюдать продуктовые ограничения:

- показываются только доступные категории;
- число показанных категорий фиксировано;
- две категории с одним `group_id` не попадают в один набор;
- вероятности назначения должны быть известны или вычислимы.

## 9. Проверки Готовности Данных

Перед моделированием синтетика должна пройти проверки:

- у каждой пары пользователь--период есть хотя бы один допустимый набор категорий;
- ни одно событие не находится после даты отсечения;
- текущий показ не попадает в историю трансформера;
- вероятности назначения положительны для поддержанных действий;
- рандомизированный показ плохо предсказывается по признакам до раздачи;
- стратегия-оракул лучше случайной стратегии на синтетической валидации;
- ранжирование по отклику не совпадает полностью с uplift-ранжированием.
