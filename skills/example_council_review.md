# Пример работы ИИ-навыка `jmlc-adaption-council`: реальный цикл ревью

Ниже — фактический цикл из **двух прогонов** навыка `jmlc-adaption-council` по этому
репозиторию с реальными находками и исправлениями между ними. Это не иллюстрация
формата: находки воспроизводимы по коду, а закрытие проверено повторным прогоном.

Формат вывода навыка задан в [`jmlc-adaption-council/SKILL.md`](jmlc-adaption-council/SKILL.md):
ревью по пяти линзам (ML, causal, product, engineering, NDA) → `Decision / Findings / Chairman Summary / Next Steps`.

---

## Прогон 1 — найдено (`Decision: needs changes`)

```markdown
Findings:

  по предраздаточным признакам), но в коде она не реализована — validate_synthetic_dataset
  возвращает только структурные проверки. Claim статьи не подкреплён кодом.
- P1 (ML): calibrate_stacked_scores — не калибровка, а ручной ансамбль с весами
  0.94/0.03/0.03; stacking почти совпадает с трансформером. Имя «calibration» вводит
  в заблуждение.
- P2 (Engineering): README называет «конфигурацией по умолчанию» 80/12/6, но дефолты CLI
  и SyntheticConfig — другие (300/18/8, 900). Голый запуск даст не headline-числа.
- P2 (Engineering): закоммичены __pycache__/.DS_Store/artifacts.
```

**Верификация находок (человеком).** Три первые подтвердились по коду. Четвёртая
(гигиена git) оказалась **ложной**: `git ls-files` пуст по этим путям, `.gitignore`
их покрывает — навык увидел файлы на диске, не сверившись с индексом. Отсеяна.

## Что исправлено

1. **Causal overlap-проверка реализована** — [`evaluate_treatment_overlap`](../utils/metrics.py)
   обучает модель предсказывать факт показа по предраздаточным признакам и считает ROC AUC
   на val/test (≈0.49 — рандомизация чистая). Вызывается в пайплайне, сохраняется в
   `propensity_overlap.csv`. Claim статьи теперь подкреплён кодом.
2. **Настоящая калибровка** — [`calibrate_stacked_scores`](../utils/models.py) переписана
   на per-arm Platt scaling: параметры подбираются на shown-строках калибровочного среза
   для головы показа и на not-shown — для головы контроля. Скоры реально меняются
   (`p1: 0.37→0.41`), Brier улучшается; имя стало честным.
3. **Терминология default** — дефолты CLI приведены к demo-конфигу `80/12/6`, в README и
   `final_demo_results.md` формулировка изменена на «demo-конфигурация (seed=42)».

## Прогон 2 — проверено (`Decision: ready`)

```markdown
Статус прошлых находок:
1. overlap-проверка: закрыта. Воспроизведено: AUC = 0.49 (val) / 0.49 (test); признаки
   без таргет-утечек; вызов в run_experiment.py, сохранение в propensity_overlap.csv.
2. calibration: закрыта. Настоящая per-arm Platt-калибровка; выполнено отдельно — скоры
   меняются (p1 0.37→0.41, p0 0.20→0.17), не no-op и не ансамбль.
3. default-терминология: закрыта. README, final_demo_results, Dockerfile, DAG и CLI
   единообразно используют demo-конфиг 80/12/6.

P0/P1: нет. Остались только косметические P2 (overlap не упомянут в одном из четырёх
docs; преимущество трансформера стоит явно пометить как дизайн синтетики). Артефакт готов.
```

Оба косметических замечания второго прогона тоже внесены (overlap добавлен в
`docs/methodology.md`; в `final_demo_results.md` явно оговорено, что превосходство
трансформера по uplift alignment — запланированное свойство синтетики, а не
самостоятельный эмпирический результат).

---

Что показывает пример: навык не заменяет автора, а **структурно проверяет артефакт по
линзам** и находит реальные проблемы (рассинхрон «claim статьи ↔ код», вводящее в
заблуждение имя функции, несоответствие документации). Верификация человеком отсеивает
ложные срабатывания, а цикл «прогон → исправления → повторный прогон» доводит артефакт
до состояния `ready`.
