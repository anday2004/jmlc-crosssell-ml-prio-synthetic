# Методология

## Постановка

Для каждой пары `user_id, period_id` система должна выбрать несколько
CrossSell-категорий из доступного преселекта. Для каждой категории известна
синтетическая ценность `NPV`. Нельзя выбрать две категории с одинаковым
`group_id`, поэтому стратегия должна не только ранжировать кандидатов, но и
собирать валидный набор.

В репозитории сравниваются две основные постановки:

- ранжирование по отклику: выбрать категории по `p(utilization) * NPV`;
- ранжирование по extra NPV: выбрать категории по `NPV * (p1 - p0)`.

Здесь `p1` означает вероятность утилизации при показе категории, а `p0` —
вероятность утилизации без показа. Разность `p1 - p0` является синтетическим
инкрементальным эффектом.

## Данные

Генератор строит четыре таблицы:

- `users`: стабильные свойства пользователя;
- `categories`: категория, `group_id`, `NPV`, признаки доступности и embedding;
- `events`: исторические клики, транзакции, действия в приложении и прошлые предложения;
- `assignments`: строка на пользователя, период и категорию.

История для моделей строится только по событиям, доступным до cutoff:

```text
event_time <= period_id * period_days - data_lag_days
```

Это правило используется и для табличных признаков, и для последовательности
событий трансформера.

## Модели

Табличная ветка использует небольшой градиентный бустинг на decision stumps.
Он реализован внутри репозитория, чтобы пример запускался без тяжелых
зависимостей.

Mini-transformer ветка строит последовательностное представление пользователя:

```text
h_i = MiniTransformer(events_i)
```

Каждое событие кодируется через эмбеддинги категориальных полей, позиционный
embedding, embedding давности и числовую интенсивность события. Encoder
обучает event embeddings, attention pooling и supervised head по shown-исходам
на train-периодах. Затем представление пользователя объединяется с embedding
категории:

```text
h_i, e_k, h_i * e_k, |h_i - e_k|
```

Поверх этих признаков обучается S-learner: одна outcome-модель получает
признаки пары пользователь--категория и флаг показа `shown_feature`. Для
оценки uplift одна и та же модель запускается дважды: при `shown_feature=1`
получается `p1`, при `shown_feature=0` получается `p0`. Для response ranking
используется прогноз `p1`.

Stacking-модель использует исходные табличные признаки и прогнозы
transformer-ветки. Это имитирует сценарий, где последовательностная модель
добавляется как сильный дополнительный признак в уже существующий контур.

## Стратегии

Слой выбора получает уже проскоренные пары `user_id, period_id, category_id`.
Дальше он жадно собирает набор:

1. взять доступную категорию с максимальным score;
2. убрать из преселекта все категории с тем же `group_id`;
3. повторить до топ-3.

Так один и тот же слой выбора можно применять к модели отклика, uplift-модели,
трансформеру и stacking-модели.

## Оценка

Новая стратегия оценивается по randomized logs на test-периоде на двух уровнях.
Основная практическая диагностика -- category-level: если выбранная стратегией
категория совпала с фактически показанной в случайной раздаче, ее вклад
взвешивается через inverse propensity score:

```text
weight = 1 / show_propensity
```

Такой уровень естественен для задачи, где ценность считается по отдельным
категориям, и дает больше совпадений между новой политикой и randomized log.
Дополнительно считается строгая set-level оценка: действием считается весь
top-3 набор, а совпадение засчитывается только если стратегия выбрала ровно тот
же unordered набор, который был случайно показан. Эта оценка шумнее, но лучше
проверяет качество политики как целого набора.

В отчете считаются:

- `IPS` и `SNIPS` на уровне категорий;
- `set_IPS` и `set_SNIPS` на уровне top-3 набора;
- `DR`: диагностика в стиле doubly robust на отложенном периоде, где модельная оценка ценности используется как baseline, а randomized log добавляет поправку по наблюдаемому исходу;
- `ESS`: effective sample size;
- бутстреп-интервалы;
- значение синтетического потолка, доступное только в синтетике.

Отдельно считаются диагностические метрики моделей:

- `ROC AUC` и `Average Precision` для фактически наблюдаемых outcome;
- `Brier score` как простая проверка вероятностного качества;
- корреляция predicted uplift с синтетическим истинным uplift;
- средний истинный extra NPV в верхнем дециле predicted extra NPV;
- `roc_auc_shown`: causal overlap-проверка — простая модель предсказывает факт показа по предраздаточным признакам; значение около `0.5` означает, что рандомизация чистая и в логах нет скрытой селекции.

Синтетический потолок нужен для sanity check: если модельные стратегии стабильно
лучше random и приближаются к потолку, значит экспериментальная среда отражает
смысл задачи.

Важно: IPS/SNIPS являются оценками по конечному randomized log. Поэтому на
маленьких синтетических запусках стратегия-потолок может не быть первой по
IPS/SNIPS из-за дисперсии. Для интерпретации нужно смотреть вместе на IPS/SNIPS,
DR, ESS, доверительные интервалы и синтетические истинные значения.
